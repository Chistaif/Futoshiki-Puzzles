\section{Experiments and Evaluation}

\subsection{Experimental Setup}
This section evaluates solver behavior on 10 puzzle instances (\texttt{input-01} to \texttt{input-10}) with board sizes from \(4 \times 4\) to \(9 \times 9\). Experimental results were recorded in \texttt{solved.csv} and visualized by \texttt{generate\_solver\_graphs.py}. The primary metrics are runtime (seconds), peak memory usage (MB, measured via \texttt{tracemalloc}), and search effort (\texttt{node\_expanded}).

To keep experiments bounded and reproducible, each solver uses a hard node-expansion cap:
\begin{itemize}
	\item \texttt{MAX\_NODES\_BRUTE\_FORCE = 100000}
	\item \texttt{MAX\_NODES\_BACKTRACKING = 3000000}
	\item \texttt{MAX\_NODES\_BACKWARD\_CHAINING = 1500000}
	\item \texttt{MAX\_NODES\_FORWARD\_CHAINING = 5000000}
	\item \texttt{MAX\_NODES\_ASTAR = 5000000}
\end{itemize}

To prevent unbounded runs, the test harness also enforces a wall-clock timeout of 180 seconds.

\subsection{Evaluation Protocol and Metrics}
\subsubsection{Input and Output Test Cases}
The test suite follows project requirements by covering multiple grid sizes and difficulty levels. Each run returns either: (i) a valid solved grid, (ii) \texttt{False}/no solution under node cap, (iii) \texttt{time\_out} at 180 seconds, or (iv) \texttt{SKIPPED} for intentionally excluded brute-force cases to avoid intractable runs on larger instances.

\subsubsection{Evaluation Metrics}
\begin{itemize}
	\item \textbf{Runtime}: elapsed execution time in seconds.
	\item \textbf{Memory Usage}: peak memory footprint in MB.
	\item \textbf{Algorithmic Complexity Proxy}: number of expanded/inferred nodes.
\end{itemize}

\subsubsection{Runtime Analysis}
Figure~\ref{fig:runtime-log} reports runtime on a logarithmic scale. Forward Chaining is consistently the fastest method across all inputs, with runtimes typically in the millisecond range and remaining below 0.2 s even on hard instances. A* remains stable and significantly slower than Forward Chaining but generally below 10 s. Backtracking solves most small and medium cases quickly but degrades sharply on difficult boards, reaching approximately 62 s on \texttt{input-07}. Backward Chaining exhibits the highest runtime variance and includes one timeout (180 s).

\paragraph{Why log scale is necessary.}
Runtime values span several orders of magnitude (from roughly \(10^{-3}\) s to \(10^2\) s). A linear y-axis would compress fast algorithms near zero and hide meaningful differences among them. The log scale preserves relative multiplicative gaps, allowing fair visual comparison between very fast and very slow solvers.

\subsubsection{Memory Usage Analysis}
Figure~\ref{fig:memory-time} shows a strong contrast in memory behavior. Forward Chaining, Backtracking, Brute Force, and A* remain memory-efficient (mostly sub-1 MB in this dataset for A* and very low MB for Forward/Backtracking/Brute Force). In contrast, Backward Chaining grows from a few MB on easy instances to more than 1.8 GB near timeout.

\begin{figure}[H]
	\centering

	\label{fig:memory-time}

\paragraph{Memory explosion in recursive logic search.}
Backward Chaining combines recursive goal expansion, memoization tables, and dynamically accumulated facts/rules during deep search. On larger, heavily constrained instances, the search tree broadens before contradictions are discovered, causing rapid growth of intermediate states and cached derivations. This is a typical \textit{memory explosion} pattern: memory grows superlinearly with search depth and branching interactions, eventually becoming the primary bottleneck before the node cap is reached.

\subsubsection{Algorithmic Complexity (Nodes Expanded)}
Node-expansion data confirms the above trends. A* solves with very low node counts (tens to hundreds) due to MRV selection plus AC-3-based pruning and admissible heuristic guidance. Forward Chaining can still be fast despite higher node counts because each propagation step is lightweight and aggressively reduces domains early. Backtracking reaches millions of expanded nodes on difficult cases (close to its cap at \(3{,}000{,}000\)). Brute Force quickly hits its cap (\(100{,}000\)) even for small boards. Backward Chaining remains moderate in node count compared with Backtracking but is expensive per node due to recursive unification and fact/rule management, which explains its high runtime and memory.

\subsection{Comparative Analysis}

\subsubsection{Performance Leader}
Based on runtime and completion reliability in \texttt{solved.csv}, \textbf{Forward Chaining} maintains the leading overall method in this benchmark with a mean runtime of \(0.027\) s. However, \textbf{SAT Solver} achieves a competitive median runtime of \(0.0092\) s and solves all instances with 100\% success, making it exceptionally strong on typical cases. It solved all 10 test cases, achieved excellent speed on most instances, and maintained reasonable memory usage (median \(0.275\) MB).

In terms of success rate on evaluated cases, Forward Chaining, A*, and SAT Solver all reached \(10/10\) solved (100\%), Backtracking reached \(9/10\) solved (one failure near node cap), and Backward Chaining reached \(9/10\) solved with one timeout at 180 s.

The explanation is consistent with Chapter 3: Forward Chaining performs continuous constraint propagation (including inequality propagation and domain pruning similar to local arc-consistency maintenance). SAT Solver benefits from advanced DPLL/CDCL algorithms that efficiently explore the CNF search space. Both methods achieve early elimination of infeasible values, reducing the effective search space.

\textbf{A*} is the second strongest practical solver: it also solved all tested cases with low node expansion because heuristic pruning (MRV + AC-3 feasibility filtering) suppresses unpromising branches.

\subsubsection{Preferred Conditions}
The recommended use conditions are:
\begin{itemize}
	\item \textbf{Brute Force / Backtracking (\(N \leq 5\))}: suitable for small boards or teaching/debugging contexts due to simple implementation and direct interpretability. Backtracking remains practical up to medium difficulty but can degrade abruptly near hard large cases.
	\item \textbf{A* Search}: preferred when a good heuristic is available and controlled search order matters. It is robust for medium-to-large structured instances because heuristic guidance reduces expansion dramatically.
	\item \textbf{Forward Chaining (\(N \geq 6\), low to moderate constraints)}: preferred when rapid constraint propagation is effective. Excels in scenarios with early domain reduction.
	\item \textbf{SAT Solver (\(N \geq 5\), any constraint density)}: recommended as a general-purpose solver for diverse puzzle sizes and densities. Outstanding median performance (0.009 s typical) and robust 100\% success rate make it a reliable choice. Best for applications where encoding complexity is acceptable.
\end{itemize}

From an implementation perspective, the main trade-offs are:
\begin{itemize}
	\item \textbf{Brute Force}: lowest implementation complexity, lowest scalability.
	\item \textbf{Backtracking}: moderate implementation complexity, strong for small-to-medium cases, unstable worst-case behavior.
	\item \textbf{Forward Chaining}: higher implementation effort (rule engine + propagation) but excellent practical balance in this project.
	\item \textbf{A*}: high implementation complexity (state modeling, admissible heuristic, consistency checks), very good search efficiency and predictable scaling.
	\item \textbf{SAT Solver}: moderate-to-high implementation complexity (CNF encoding + SAT solver integration) but exceptional runtime performance (median 0.0092 s) and robust 100\% success rate, making it an outstanding general-purpose choice despite encoding overhead.
\end{itemize}
