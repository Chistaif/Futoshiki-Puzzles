\subsection{GUI}
\subsubsection{Overview}
The GUI is implemented with \texttt{pygame} to support end-to-end interaction: selecting inputs, playing manually, running AI solvers, and viewing results. The interface is modular, with reusable components and separate screen states, so gameplay logic and rendering remain easy to maintain.

\subsubsection{Screen Flow}
The GUI follows a simple state machine:
\begin{itemize}
    \item \textbf{Menu}: Play, AI Solver, Exit.
    \item \textbf{Selection screens}: choose board size and input file.
    \item \textbf{Game screen}: play manually or run solver visualization.
\end{itemize}

\subsubsection{Gameplay Interface}
The gameplay screen renders the puzzle grid, keeps clue cells fixed, and validates moves in real time. Main controls are:
\begin{itemize}
    \item click to select a cell;
    \item keys \(1\) to \(N\) to fill values;
    \item Backspace/Delete/0 to clear;
    \item timer for the solving session.
\end{itemize}
Inequality constraints are displayed directly between cells, and completion is announced when all constraints are satisfied.

\subsubsection{AI Solver Visualization}
AI visualization is handled by \texttt{AISolverManager} using a worker thread and step queue, so the UI stays responsive while the solver runs. Solver steps are streamed to the board incrementally, enabling users to observe search behavior instead of only final output.

Key behaviors include:
\begin{itemize}
    \item highlight of current solver actions;
    \item visual distinction for backtracking steps;
    \item ability to skip remaining animation;
    \item final validation before showing solved state.
\end{itemize}

\subsubsection{Visual Design}
Visual styling is intentionally simple and consistent across screens to prioritize readability of board states and AI progress. Reusable components help keep interaction patterns uniform and reduce implementation complexity.

Overall, the GUI acts as a practical bridge between puzzle data, user interaction, and AI solver execution.
